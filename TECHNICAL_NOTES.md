# GroceryGPT — Full Technical Documentation

---

## 1. Project Overview

GroceryGPT is a fully local, end-to-end Retrieval-Augmented Generation (RAG) application that lets
you ask natural language questions about UK grocery products and receive answers grounded in real
product data, with cited sources and RAGAS evaluation metrics. Everything runs on your machine with
zero paid API calls.

Built as a portfolio piece for a Data Scientist role. Demonstrates:
- API data ingestion at scale with retry logic
- Dense vector embeddings with Apple MPS acceleration
- ChromaDB vector store with cosine similarity search
- LangChain RAG chain backed by a local LLM
- Rigorous evaluation using the RAGAS framework (fully local)
- Production-quality code: type hints, modular structure, Makefile pipeline, pytest suite

## 4. How Each Component Works

### 4.1 Data Ingestion — src/ingest.py

Fetches UK grocery products from the Open Food Facts v2 REST API.

Endpoint pattern:
```
GET https://world.openfoodfacts.org/api/v2/search
    ?countries_tags=united-kingdom
    &fields=code,product_name,brands,categories,ingredients_text,
            nutriments,quantity,nutriscore_grade,nova_group,allergens,labels,stores
    &page_size=100
    &page={n}
```

Behaviour:
- Paginates 100 products per page, sleeping 0.3 seconds between requests (polite crawling)
- Sets User-Agent: GroceryGPT/1.0 (educational) as required by OFF terms of service
- Filters out rows where product_name or ingredients_text are empty
- Deduplicates on code (the product barcode)
- Merges new results with any existing parquet file so repeated runs accumulate data
- Saves to data/products.parquet

The text column fed to the embedder is a structured template:
```
Product: Digestive Biscuits with Wheatgerm
Brand: Tesco
Categories: Biscuits, Snacks
Ingredients: Wholemeal Wheat Flour, Vegetable Oil, Sugar, Oatmeal...
Allergens: en:gluten, en:milk
Labels: Vegetarian
Nutri-Score: d
NOVA group: 4
Quantity: 400g
```

This structured format ensures the embedding captures all product facets (ingredients, allergens,
brand, nutrition) rather than just the product name.

CLI usage:
```bash
python -m src.ingest --limit 2000
```

### 4.2 Vector Index — src/build_index.py

Converts all product text columns into 384-dimensional dense vectors and stores them in ChromaDB.

Steps:
1. Reads data/products.parquet
2. Detects device: MPS (Apple Silicon GPU) then CUDA then CPU
3. Loads BAAI/bge-small-en-v1.5 via SentenceTransformer
4. Initialises ChromaDB PersistentClient at data/chroma/ with telemetry disabled
5. Drops and recreates the products collection on every run (idempotent)
6. Embeds in batches of 64 with normalize_embeddings=True so vectors lie on the unit sphere
7. Stores: id=barcode code, document=text string, embeddings=float list, metadata={product_name, brands, categories, nutriscore_grade}

With normalised vectors, cosine similarity becomes equivalent to a dot product, which is faster
and what ChromaDB's HNSW index uses internally with the cosine distance metric.

CLI usage:
```bash
python -m src.build_index
```

### 4.3 RAG Chain — src/rag.py

The GroceryRAG class exposes two public methods.

**retrieve(query: str, k: int) -> list[dict]**

1. Embeds the query string with the same sentence-transformer model
2. Calls ChromaDB collection.query(query_embeddings=[...], n_results=k)
3. Returns a list of dicts: {text, metadata, distance}
   - distance is cosine distance: 0 = identical, 2 = opposite directions

**answer(query: str, k: int) -> dict**

1. Calls retrieve() to get top-k product texts
2. Joins them with separator lines into a context block
3. Fills a strict prompt template (see below)
4. Sends to ChatOllama(model="llama3.2:3b", temperature=0.1) — low temperature for factual consistency
5. Returns {answer: str, sources: list[dict], contexts: list[str]}

Prompt template:
```
You are a UK grocery assistant. Answer the user's question using ONLY the
product information in the context below. If the answer is not in the context,
say "I don't have that information in my product catalogue." Be concise.
Cite product names you reference in square brackets.

Context:
{context}

Question: {question}

Answer:
```

The ONLY instruction grounds the model in the retrieved context and prevents hallucination.
The citation instruction (square brackets) makes source attribution visible in answers.

CLI usage:
```bash
python -m src.rag --query "What allergens are in digestive biscuits?" --k 5
```

### 4.4 Evaluation — src/evaluate.py

Runs the RAGAS evaluation framework entirely locally.

RAGAS configuration:
- LLM judge: ChatOllama wrapped in LangchainLLMWrapper, passed to ragas.llms
- Embedding judge: HuggingFaceEmbeddings(BAAI/bge-small-en-v1.5) wrapped in LangchainEmbeddingsWrapper, passed to ragas.embeddings

For each of the 25 test questions:
1. Calls rag.answer() to get the generated answer and retrieved contexts
2. Pairs with the ground_truth string from test_questions.json
3. Builds a HuggingFace Dataset row: {question, answer, contexts, ground_truth}

RAGAS then runs four metrics over this dataset:
- faithfulness — are all claims in the answer supported by the retrieved context?
- answer_relevancy — does the answer directly address the question?
- context_precision — what fraction of retrieved contexts are actually relevant?
- context_recall — does the context contain all information needed to answer the question?

Outputs:
- results/eval.json — per-question scores (gitignored)
- results/eval.md — overall means and per-category breakdown (committed to git)
- stdout — summary table

CLI usage:
```bash
python -m src.evaluate
```

### 4.5 Streamlit UI — app.py

- @st.cache_resource loads GroceryRAG once on startup; subsequent reruns reuse the same object
- Chat history stored in st.session_state.messages as a list of role/content dicts
- Sidebar: shows model name, top-k slider (1 to 10, default 5), reset chat button
- On user submit: calls rag.answer(query, k=top_k), renders the answer with st.markdown
- Sources appear in an st.expander showing product name and similarity score = 1 - cosine_distance

### 4.6 Test Suite — tests/test_rag.py

Three pytest sanity tests with a module-scoped GroceryRAG fixture (loaded once for all tests):

1. test_collection_not_empty
   Opens ChromaDB directly. Asserts collection.count() > 0.
   Fails loudly if build_index has not been run.

2. test_retrieve_returns_k_results
   Calls rag.retrieve("biscuits", k=3). Asserts len(results) == 3.
   Verifies the retriever returns exactly the requested number of results.

3. test_answer_returns_non_empty_string
   Calls rag.answer("What allergens are in a popular biscuit brand?").
   Asserts the answer field is a non-empty string.
   Requires Ollama to be running.

## 5. Evaluation Methodology

### 5.1 Test Set Construction

File: eval/test_questions.json — 25 hand-crafted question-answer pairs.

Each entry:
```json
{
  "question": "Are there any products that contain peanuts?",
  "ground_truth": "Some products such as peanut butter, certain chocolate bars, and cereal bars contain peanuts and may list them as an allergen.",
  "category": "allergen_filter"
}
```

Seven categories:

| Category | Questions | What it tests |
|----------|-----------|---------------|
| ingredient_lookup | 5 | Asks what a product contains |
| allergen_filter | 5 | Queries for products safe for dietary restrictions |
| brand_recommendation | 4 | Asks about specific UK brands (Walkers, Heinz, Cadbury, Tesco) |
| nutriscore_query | 4 | Asks about nutritional quality scores |
| vegan_vegetarian | 3 | Asks about plant-based or animal-free products |
| category_browsing | 4 | Browses product types (pasta, oils, bread, organic) |
| edge_case | 1 | Queries for a product that does not exist (expects "I don't have that information") |

Ground truths are phrased generically to match realistic OFF data and avoid hardcoding specific
product barcodes. They describe the kind of answer that would be correct rather than one exact string.

### 5.2 RAGAS Metric Definitions

**Faithfulness**
Breaks the generated answer into atomic claims, then checks whether each claim is entailed by the
retrieved contexts using the LLM as a judge. Score = supported_claims / total_claims.
A faithfulness of 1.0 means every claim in the answer is backed by the retrieved product data.
Low faithfulness means the LLM is hallucinating facts not present in the context.

**Answer Relevancy**
Generates several synthetic questions from the answer, then measures the cosine similarity between
those synthetic questions and the original question. High score means the answer directly addresses
what was asked. Low score means the answer is verbose, off-topic, or evasive.

**Context Precision**
Checks whether the retrieved contexts that were actually useful for the answer appear near the top
of the ranked list. Measures retrieval precision — are we surfacing the right products first?

**Context Recall**
Checks whether each sentence of the ground truth answer can be attributed to at least one of the
retrieved contexts. Measures retrieval coverage — does the retriever find all the information needed?

### 5.3 Actual Results

| Metric | Score | Coverage | Method |
|--------|-------|----------|--------|
| context_recall | 0.4500 | 25/25 | NLI sentence-level entailment |
| faithfulness | 0.5000 | 5/25 | LLM structured JSON output |
| answer_relevancy | 0.7486 | 4/25 | LLM structured JSON output |
| context_precision | N/A | 0/25 | LLM structured JSON output |

Context_recall is the most reliable because it uses sentence-level Natural Language Inference rather
than asking the LLM to produce structured JSON. The other three metrics require the judge LLM to
output a valid JSON object for each sub-question. llama3.2:3b (a 3 billion parameter model) frequently
either times out (takes longer than 30 seconds per sub-prompt) or returns free-text prose instead of
valid JSON, causing those evaluations to fail silently and record NaN.

Per-category context_recall breakdown:
- edge_case: 1.00 (correctly returned "I don't have that information")
- allergen_filter: 0.70 (strong signal in allergen fields)
- vegan_vegetarian: 0.50
- category_browsing: 0.375
- brand_recommendation: 0.375
- ingredient_lookup: 0.3125
- nutriscore_query: 0.25 (weakest; LLM hedges rather than citing specific Nutri-Score values)

## 7. Errors Encountered and How They Were Fixed

### Error 1: Open Food Facts API returning HTTP 503

**When it happened:** During the first and second ingest runs.

**Symptom:** After fetching page 1 (100 products), every subsequent page returned
503 Service Temporarily Unavailable and the script exited early with only 100 products saved.

**Root cause:** The OFF API rate-limits crawlers and occasionally has transient server issues.
The original code used requests.get() with no retry logic and broke out of the pagination loop on
any exception.

**Fix:** Extracted a _fetch_page() helper with exponential back-off:

```python
def _fetch_page(url: str, max_retries: int = 4) -> list[dict]:
    delay = 2.0
    for attempt in range(max_retries):
        response = requests.get(url, headers=_HEADERS, timeout=30)
        if response.status_code in (429, 500, 502, 503, 504):
            wait = delay * (2 ** attempt)   # 2s, 4s, 8s, 16s
            time.sleep(wait)
            continue
        response.raise_for_status()
        return response.json().get("products", [])
    return []   # give up after max_retries, continue to next page
```

Also added a merge strategy so repeated ingest runs accumulate products:

```python
if out_path.exists():
    existing = pd.read_parquet(out_path)
    merged = pd.concat([existing, df], ignore_index=True)
    merged = merged.drop_duplicates(subset=["code"]).reset_index(drop=True)
    df = merged
```

Result: 4 ingest runs accumulated 1,891 unique products despite the API being intermittently
unavailable. The app's title says 2,000+ but 1,891 is what survived deduplication.

### Error 2: ModuleNotFoundError: No module named 'src' in pytest

**When it happened:** First run of pytest tests/.

**Symptom:**
```
ImportError while importing test module 'tests/test_rag.py'
ModuleNotFoundError: No module named 'src'
```

**Root cause:** pytest discovers and imports test files but does not automatically add the project
root to sys.path unless a pyproject.toml or setup.py registers the package. Without this, the
`from src.config import ...` import inside test_rag.py cannot find the src package.

**Fix:** Two files added:

tests/__init__.py (empty) — makes the tests directory a proper Python package so pytest treats
it consistently.

conftest.py (at project root):
```python
import sys
from pathlib import Path
sys.path.insert(0, str(Path(__file__).parent))
```

pytest automatically loads conftest.py before any tests, so by the time test_rag.py is imported,
the project root is already on sys.path and `from src.config import ...` resolves correctly.

### Error 3: Ollama Connection Refused in pytest (transient)

**When it happened:** test_answer_returns_non_empty_string on first pytest run after fixing Error 2.

**Symptom:**
```
httpx.ConnectError: [Errno 61] Connection refused
```

**Root cause:** Ollama was running (confirmed immediately after with
`curl http://localhost:11434/api/tags` which returned the model list). The error was a transient
timing issue — Ollama was in a brief unresponsive state at the moment the module-scoped
GroceryRAG fixture established its connection. Because the fixture is module-scoped, it connects
once at test collection time, which is the worst possible moment if the server is mid-operation.

**Fix:** No code change was needed. The CLI test `python -m src.rag --query "..."` ran
successfully, confirming the pipeline worked. Re-running pytest immediately produced 3 passed.

### Error 4: RAGAS TypeError — float() argument must be a string or a real number, not 'list'

**When it happened:** At the end of the 16-minute RAGAS evaluation, after all 100 metric
sub-evaluations completed.

**Symptom:**
```python
"faithfulness": float(result["faithfulness"])
TypeError: float() argument must be a string or a real number, not 'list'
```

**Root cause:** RAGAS changed its return format between versions. In RAGAS < 0.2, result["faithfulness"]
returned a single aggregate float. In RAGAS 0.4.3 (what pip installed), it returns a Python list with
one float per evaluated question, where questions that failed to score come back as None.

**Fix:** Replaced the direct float() cast with a helper that handles both formats and also
filters NaN values (which Python's `v is not None` check would not catch):

```python
def _mean_score(val: Any) -> float:
    import math
    if isinstance(val, (list, tuple)):
        valid = [v for v in val if v is not None
                 and not (isinstance(v, float) and math.isnan(v))]
        return float(sum(valid) / len(valid)) if valid else float("nan")
    return float(val)
```

### Error 5: Most RAGAS Metrics Returning nan

**When it happened:** After fixing Error 4, the evaluation completed but printed:
```
faithfulness      : nan
answer_relevancy  : nan
context_precision : nan
context_recall    : 0.4500
```

**Root cause:** llama3.2:3b is a 3-billion-parameter quantised model. RAGAS sends judge prompts
asking it to output valid JSON objects like {"verdict": 1} or {"statements": [...], "verdicts": [...]}.
The 3B model frequently either:
- Timed out (took longer than 30 seconds per sub-prompt)
- Returned English prose instead of JSON (e.g. "The context was useful because...")
- Returned malformed JSON with unescaped quotes

Every failed sub-evaluation set that question's metric score to None. When averaging a list where
most values are None or NaN, the result is NaN.

context_recall avoided this problem entirely because it uses a different internal mechanism:
sentence-level Natural Language Inference (checking whether each ground truth sentence is
entailed by the retrieved contexts) rather than asking the LLM to output structured data.

**Fix:** Instead of re-running the expensive 16-minute evaluation, the per-question scores were
already saved to results/eval.json. A small post-processing script recomputed the mean from
the saved data, filtering out NaN properly, and also computed coverage (how many of the 25 questions
actually produced a valid score for each metric). The eval.json and eval.md were updated
in-place with the corrected figures.

Final honest reporting with coverage:

| Metric | Score | Coverage |
|--------|-------|----------|
| faithfulness | 0.5000 | 5/25 |
| answer_relevancy | 0.7486 | 4/25 |
| context_precision | N/A | 0/25 |
| context_recall | 0.4500 | 25/25 |

This is documented in the README with a note that llama3.1:8b would give full coverage.

### Error 6: ModuleNotFoundError: No module named 'torchvision' on Streamlit launch

**When it happened:** Running make app for the first time.

**Symptom:**
```
File "transformers/models/zoedepth/image_processing_zoedepth.py", line 43
    from torchvision.transforms.v2 import functional as tvF
ModuleNotFoundError: No module named 'torchvision'
```

**Root cause:** transformers 5.8.1 (the version pip resolved to) includes a ZoeDepth
depth-estimation model whose image processing module imports torchvision unconditionally at
Python import time — not lazily, not inside a function, but at the module level. When Python
imports the transformers package top-level namespace (which happens when sentence_transformers
imports transformers when GroceryRAG initialises), it triggers this module, which needs torchvision
even though ZoeDepth has nothing to do with grocery products.

torchvision was not in the original requirements.txt because it is not a direct dependency of
any code we wrote — it is a transitive dependency that became mandatory due to the transformers
version pip selected.

**Fix:** Installed torchvision and pinned it in requirements.txt:
```
torchvision>=0.19.0
```

The Streamlit app launched successfully on the second attempt.

## 9. Limitations and Suggested Next Steps

| Limitation | Detail | Suggested Fix |
|------------|--------|---------------|
| 1,891 products | Full OFF UK dataset is ~800k products | Nightly sync with the full OFF data dump; distributed indexing |
| No query rewriting | Complex queries like "something low-fat for kids" may not retrieve well | Add HyDE (hypothetical document embeddings) or LLM-based query expansion |
| No re-ranking | Retrieval is flat cosine similarity with no cross-encoder refinement | Add cross-encoder/ms-marco-MiniLM-L-6-v2 re-ranker after retrieval |
| Single-turn only | Each question is independent; no conversation memory | Add LangChain ConversationBufferMemory for multi-turn context |
| RAGAS coverage | 3B model fails structured-output metrics | Re-run evaluate with llama3.1:8b for full coverage |
| No fine-tuning | Embeddings are generic, not grocery-specific | Fine-tune BAAI/bge-small-en-v1.5 on grocery query-product pairs |
| Ollama latency | ~3-8s per query on MPS with 3B model | Use quantised GGUF on a faster GPU or switch to llama3.2:1b for demo |
| No metadata filters | ChromaDB supports pre-filtering by metadata fields | Add filter controls in the Streamlit sidebar (e.g. Nutri-Score, brand) |
